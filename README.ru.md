# add-reasoning-to-prs

**Самодокументирующиеся PR.** Хук для [Claude Code](https://docs.claude.com/en/docs/claude-code), который автоматически записывает *почему* — решения, компромиссы, допущения, ограничения — в каждый pull request. Составляется вашим собственным агентом, на вашей собственной машине.

> Переводы — это добросовестное усилие сообщества, и они могут отставать. Авторитетным источником является английская версия ([README.md](README.md)).

ИИ пишет ваш код. А *почему* — не пишет никто. Через три недели ваша история Git — это
груда решений, которые формально ваши, но которые вы на самом деле никогда не принимали —
diff говорит, *что* изменилось, `git blame` говорит, *кто*, а обоснование просто…
исчезло.

Это хук, который чинит это в источнике: прямо перед тем, как ваш агент откроет PR (или
сделает коммит напрямую в ветку по умолчанию), он записывает короткий **блок «почему»** —
принятые решения, взвешенные компромиссы, сделанные допущения и то, что было сознательно
оставлено — прямо в описание. Из реальной сессии. Не из diff.

## Установка

Одна команда:

```sh
npx add-reasoning-to-prs
```

Она копирует автономный хук без зависимостей в стабильное место и регистрирует его в
настройках вашего Claude Code (`~/.claude/settings.json`) как хук `PreToolUse`. Без шага
сборки, без аккаунта, без конфигурации.

Либо установите **плагин Claude Code** из маркетплейса (рекомендуется — он регистрирует
хук из манифеста плагина и обновляется вместе с ним, не трогая настройки вашего проекта).

> **Требования:** Claude Code и Node.js ≥ 22.18. Пока только Claude Code — Cursor и Codex
> на очереди.

## Что он делает

| | |
|---|---|
| **Автоматический блок «почему» при создании PR** | Когда ваш агент выполняет `gh pr create` (или делает коммит в ветку по умолчанию) без него, хук сначала просит написать блок, а затем повторно запустить команду. Ноль ручных шагов после установки. |
| **Только вперёд — никогда не пересказывает diff** | Он фиксирует то, что diff *не может* показать: рассуждения и сознательно принятые риски. Он начинает с единственного пункта, который рецензент не смог бы получить из кода, и никогда не добавляет воды вроде «отрефакторил X, улучшил Y». |
| **Локально, на вашей собственной подписке, без аккаунта** | Блок составляется вашим собственным агентом, внутри сессии. Без обращения к серверу, ничего не сохраняется, и ваш исходный код не покидает вашу машину. |
| **Никогда не выдумывает** | Каждая строка должна восходить к реальному решению из сессии. Если агент на самом деле не рассуждал, блок не добавляется — для рутинного изменения пустой блок и есть правильный ответ. |

## Чего он **не** делает

- **Это не бот-ревьюер.** Он не оценивает ваш код, не выставляет баллы PR и не блокирует слияние. Он находится *над diff*, рядом с ревью — он добавляет контекст, а не судит.
- **Это не диаграмма, не вики и не граф знаний.** Ничего не нужно просматривать, ничего не нужно держать в синхронизации. Только «почему», записанное там, куда рецензенты и так смотрят: в описании PR.
- **Он не читает и никуда не отправляет ваш исходный код.** Без аккаунта, без загрузки, без телеметрии. Блок составляется на вашей машине агентом, который у вас уже запущен, на вашей собственной подписке к модели.
- **Он только вперёд.** Он пишет «почему» для PR, идущих дальше. Он никогда не переписывает вашу закрытую историю и никогда не трогает вашу команду git, если что-то пойдёт не так — ошибка хука всегда завершается безопасно (fail-open).

**Бесплатно и под MIT, навсегда.** [Backthread](https://backthread.dev) — платный
размещённый слой — делает то, что локальный хук по своей структуре не может: межкомандную,
историческую и проактивную доставку — «почему», которое приходит к вам, с поиском по всей
вашей кодовой базе, по агентам всех участников. Этому хуку ничего из этого не нужно, чтобы
быть полезным самому по себе. Если однажды вам понадобится командный вид, он есть на
[backthread.dev](https://backthread.dev).

## Как это работает

Хук следит за двумя моментами: открытием PR (`gh pr create`) и прямым коммитом в ветку по
умолчанию. Увидев один из них без блока «почему», он просит вашего агента составить
обоснованный, направленный только вперёд блок из собственных рассуждений сессии — **Решения,
Компромиссы, Допущения и Ограничения** за изменением — и повторно запустить команду. Блок
обёрнут в невидимый маркер, поэтому пишется один раз и никогда не дублируется.

- **Коммиты в feature-ветке уступают дорогу PR.** Работа, разбросанная по нескольким
  сессиям, переносится вперёд локально, так что блок PR охватывает всю ветку — даже если её
  открывает другая сессия.
- **Ничего никогда не выдумывается.** Агент сначала делает быструю самопроверку и отбрасывает
  любую строку, которую не может возвести к реальному решению; если сессия не рассуждала, блок
  не добавляется.
- **Он никогда не блокирует вашу команду git.** Любой режим отказа — это тихий no-op — в
  худшем случае блок просто не добавляется.

Каждый блок несёт небольшую видимую атрибуцию, чтобы рецензенты видели, откуда он взялся — а
вы можете свободно его редактировать или удалить.

## Управление

- **Отключить для репозитория:** `git config add-reasoning-to-prs.disabled true`
- **Пропустить один коммит/PR:** поместите `[skip-why]` в любое место команды.
- **Отключить глобально:** задайте `ADD_REASONING_TO_PRS_DISABLE=1` в окружении, из которого
  вы запускаете Claude Code.

## Дорожная карта

Честное состояние проекта — что работает, что дальше и чего он пока не делает.

- **Работает сегодня:** блок «почему» при `gh pr create`, откат к сообщению коммита при
  прямом push, перенос между несколькими сессиями по всей ветке, 100% локально (ваша
  собственная модель, без аккаунта), никогда не выдумывает и завершается безопасно.
- **Дальше:** поддержка Cursor и Codex (пока только Claude Code — это первое в списке) ·
  охват PR, открытых в браузере · более компактный формат блока «почему».
- **Известные пробелы:** сбор между сессиями делается по мере возможности (локальный
  черновик на каждую ветку), а то, как читается блок после squash-merge, всё ещё дорабатывается.

Полный список — в [Issues](../../issues) и [Discussions](../../discussions) — ставьте 👍
ограничениям, которые важны именно вам, и они поднимутся выше в списке.

## Участие

Вклад приветствуется — особенно исправления багов, более точные тексты промптов/подсказок,
охват краевых случаев и поддержка новых агентов. Это маленький инструмент с единственным
назначением, и он намерен таким и остаться, поэтому запросы на расширение охвата обычно
отклоняются (вежливо и с обоснованием). Начните с
[руководства для контрибьюторов](CONTRIBUTING.md) и метки
[`good first issue`](../../labels/good%20first%20issue); если вы не уверены, вписывается ли
идея, сначала откройте [Discussion](../../discussions).
